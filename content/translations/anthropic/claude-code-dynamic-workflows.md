今天，我们在 Claude Code 中推出**动态工作流**。借助它，Claude 可以端到端地承担更具挑战性的任务：过去往往需要按季度规划的工作，现在有机会在几天内完成。

在一次会话中，Claude 会根据任务动态编写编排脚本，启动数十到数百个并行子代理，并在把结果交给你之前先完成检查和校验。

有些问题本来就不适合交给单个代理一次性解决，尤其是在复杂的遗留代码库里。比如：

- 在整个服务范围内追查一个隐蔽 bug；
- 完成一场涉及数百个文件的迁移；
- 在真正投入实施前，从多个角度压力测试一份方案。

动态工作流就是为这类任务准备的。它可以把这些工作从规划、执行到验证完整接住。

目前，动态工作流已作为研究预览版开放，支持：

- Claude Code CLI；
- Claude Desktop；
- VS Code 扩展；
- Max、Team 和 Enterprise 方案用户（Enterprise 需管理员启用）；
- Claude API；
- Amazon Bedrock；
- Vertex AI；
- Microsoft Foundry。

> 注意：动态工作流消耗的 token 可能明显高于普通 Claude Code 会话。建议先从边界清晰的小任务开始，熟悉它在你实际工作流中的用量表现。

为了获得最佳体验，使用动态工作流时建议开启 **auto mode**。之后，你有两种方式可以启动工作流：

1. 直接让 Claude 创建动态工作流，例如输入：`Create a workflow`；
2. 开启 Claude Code 专属的新设置：`ultracode`。

`ultracode` 可以通过 effort 菜单访问。它会把 effort level 设为 `xhigh`，并让 Claude 自动判断什么时候应该为当前任务启用工作流。

## 动态工作流能做什么

Anthropic 内部团队和早期体验用户已经把动态工作流用于许多场景，包括：

### 全代码库 bug 排查、基于性能分析的优化审计，以及安全审计

Claude 可以并行搜索整个服务或代码仓库，再对每一条发现进行独立验证，确保最终报告中呈现的是真问题。

同样的模式也适用于代码加固：比如在整个代码库中检查认证逻辑、输入校验，以及潜在的不安全实现。

### 大规模迁移与现代化改造

Claude 可以端到端地处理框架替换、废弃 API 迁移，甚至跨越数千个文件的语言移植。

### 需要反复确认的关键工作

当错误答案的代价很高时，工作流可以让 Claude 从多个独立路径尝试解决问题，并安排“对抗性代理”主动挑战和推翻已有结论。你最终看到的，是经过多轮质疑和收敛后的版本。

> “动态工作流在大型代码库的发现与审查任务中尤其有价值。我们已经看到，它能识别死代码，并发现传统静态分析遗漏的清理机会，帮助工程师更快推进维护和重构工作。”
>
> —— Alessio Vallero，高级工程经理

> “动态工作流填补了单个子代理和完整代理团队之间的空白。从计划到实现的过程变得很顺畅，所以即使任务运行很久，我们也能在不失去可见性的前提下信任它。”
>
> —— Ken Takao，首席系统工程师

## 用动态工作流重写 Bun

最近 Bun 的重写，是动态工作流在大规模工程任务上的一个典型案例。

Jarred Sumner 使用动态工作流将 Bun 从 Zig 移植到 Rust。最终，现有测试套件通过率达到 99.8%，生成了约 75 万行 Rust 代码，并且从首次提交到合并只用了 11 天。

整个过程由多个工作流分阶段完成：

1. 第一个工作流为 Zig 代码库中每个结构体字段映射合适的 Rust 生命周期；
2. 第二个工作流为每个 `.zig` 文件编写行为一致的 `.rs` 移植版本；
3. 数百个代理并行工作，每个文件都由两个审查代理把关；
4. 随后，一个修复循环持续驱动构建和测试，直到两者都干净通过；
5. 移植落地后，一个夜间工作流又处理了不必要的数据拷贝，并为每项改动打开 PR，供人工做最终审查。

这套 Rust 版本目前尚未进入生产环境，但上述过程已经全部由动态工作流完成。Jarred 之后会写更多文章介绍这件事。

## 它是如何工作的

当一个工作流启动时，Claude 会根据你的提示动态制定计划，把任务拆成多个子任务，并把工作分发给并行运行的子代理。

子代理的结果不会直接合并，而是会先经过检查。最终，你拿到的是一个统一协调后的答案。

整个过程大致如下：

- 一些代理从不同角度分析问题；
- 另一些代理尝试反驳或推翻它们的发现；
- 工作流持续迭代，直到答案逐渐收敛。

这也是动态工作流能够达到单次运行难以达到的结果的原因。

动态工作流特别适合并行、长时间运行的任务。它可以持续数小时甚至数天，处理过去可能需要数周才能完成的复杂工程工作。

运行过程中，进度会被持续保存。因此，如果任务中途被打断，它可以从中断处继续，而不是从头开始。

由于协调过程发生在对话之外，即使任务规模很大，整体计划也能保持稳定，不会随着上下文膨胀而偏离方向。

需要再次说明的是：动态工作流的用量会明显高于普通 Claude Code 会话。

第一次触发工作流时，Claude Code 会展示即将运行的内容，并请求你确认。组织管理员也可以通过托管设置关闭工作流能力。

## 如何开始

如果你使用的是 Max 或 Team 方案，或者通过 API 使用 Claude Code，动态工作流默认已经开启。

你可以直接让 Claude 创建工作流，也可以开启 Claude Code 专属设置 `ultracode`。

如果你使用的是 Enterprise 方案，动态工作流在发布初期默认关闭。管理员可以在 Claude Code 设置中轻松启用。

更多信息请阅读官方文档。

# 官方文档补充：动态工作流

> 这一节对应原文末尾 “Read the documentation to learn more” 中提到的 documentation，也就是 Claude Code 的动态工作流文档。

动态工作流目前仍处于研究预览阶段。它要求 Claude Code 版本不低于 `v2.1.154`，并面向所有付费方案开放；同时也支持 Anthropic API、Amazon Bedrock、Google Cloud Vertex AI 和 Microsoft Foundry。Pro 用户可以在 `/config` 中的 **Dynamic workflows** 一行手动开启。

简单说，动态工作流是一段 JavaScript 脚本，用来大规模编排子代理。Claude 会根据你描述的任务生成这段脚本，运行时则在后台执行。这样一来，即使工作流正在并行调度大量代理，你当前的会话依然可以保持响应。

当一个任务已经超出单轮对话能稳定协调的范围，或者你希望把编排过程固化成一段可阅读、可复用的脚本时，就适合使用工作流。典型场景包括：

- 扫描整个代码库里的 bug；
- 迁移 500 个文件；
- 对研究问题进行多来源交叉验证；
- 在决定采用某个方案前，让 Claude 从多个独立角度先起草和比较。

这篇文档主要说明四件事：

- 什么时候应该用工作流，而不是普通子代理或 skills；
- 如何运行内置工作流 `/deep-research`；
- 如何让 Claude 为你的任务编写工作流并保存下来；
- 工作流如何运行，以及如何管理运行中的工作流。

## 什么时候该使用工作流

子代理、skills 和工作流都可以处理多步骤任务。它们真正的区别在于：**计划由谁保存，又由谁推进**。

| 能力 | 子代理 | Skills | 工作流 |
| --- | --- | --- | --- |
| 本质 | Claude 启动的工作代理 | Claude 遵循的一组指令 | 运行时执行的一段脚本 |
| 谁决定下一步 | Claude 一轮一轮决定 | Claude 按提示词执行 | 脚本决定 |
| 中间结果放在哪里 | Claude 的上下文窗口 | Claude 的上下文窗口 | 脚本变量 |
| 哪些东西可以复用 | 子代理定义 | 指令本身 | 整套编排逻辑 |
| 规模 | 每轮少量委派任务 | 与子代理类似 | 每次运行可调度数十到数百个代理 |
| 中断后怎么办 | 通常要重新开始当前轮 | 通常要重新开始当前轮 | 同一会话内可以恢复 |

工作流的关键变化，是把“计划”从对话里搬进代码里。

使用子代理或 skills 时，Claude 仍然是编排者：它需要一轮一轮决定下一步该启动什么，所有结果也都会进入 Claude 的上下文。工作流则不同：循环、分支和中间结果都由脚本自己维护，Claude 的上下文里只需要接收最终答案。

把计划写进代码之后，工作流不仅能启动更多代理，还能稳定复用一套质量控制模式。比如，它可以让不同代理在结论上报前相互进行对抗式审查；也可以让多个代理从不同角度起草方案，再比较它们的优劣。最终得到的结果，通常会比一次性回答更可靠。

## 运行内置工作流

想最快体验工作流，可以运行 Claude Code 内置的 `/deep-research`。

这是一个用于多来源调研的工作流。它会在后台按阶段启动多个代理，你的会话仍然可以继续使用；等任务完成后，你会得到一份整合后的报告，而不是一长串逐轮对话记录。

### 运行工作流

使用 `/deep-research` 加上你想调研的问题即可。例如：

```text
/deep-research What changed in the Node.js permission model between v20 and v22?
```

它会从多个角度发起网页搜索，抓取并交叉检查找到的来源，最后合成为一份带引用的报告。

### 允许工作流运行

Claude Code 会询问你是否允许启动这个工作流。选择 **Yes** 后继续。具体提示会根据你的权限模式略有不同。

### 查看进度

工作流启动后会在后台运行。你可以执行：

```text
/workflows
```

然后用方向键选择对应的运行记录，按 Enter 打开进度视图。

进度视图会展示每个阶段的代理数量、token 总量和耗时。你还可以进入某个阶段，查看其中每个代理做了什么、发现了什么。

你也可以从输入框下方的任务面板查看进度。运行过程中，那里会显示一行简短状态。按向下箭头聚焦，再按 Enter 可以展开详情。

### 阅读报告

工作流完成后，报告会回到当前会话中。报告会标明每个论断来自哪些来源；没有通过交叉验证的内容，会在最终报告中被过滤掉。

如果你想把类似流程用于自己的任务，可以让 Claude 写一个工作流。等它运行效果符合预期后，还可以把它保存成自己的命令。

## 内置工作流

Claude Code 目前内置了 `/deep-research`：

| 命令 | 作用 |
| --- | --- |
| `/deep-research <question>` | 从多个角度搜索一个问题，抓取并交叉检查来源，对论断进行投票筛选，最后返回一份带引用的报告。没有通过交叉验证的论断会被过滤掉。需要可用的 `WebSearch` 工具。 |

你自己保存的工作流也会以命令形式出现，并和内置命令一样出现在自动补全里。

## 观察运行中的工作流

工作流在后台运行，因此代理工作时，会话仍然保持可用。你可以随时运行：

```text
/workflows
```

查看正在运行和已经完成的工作流，并选择其中一个打开进度视图。

进度视图会列出每个阶段的代理数量、token 总数和耗时。底部会显示可用快捷键：

| 按键 | 操作 |
| --- | --- |
| `/` | 选择某个阶段或代理 |
| Enter | 进入选中的阶段；再进入某个代理后，可以查看它的提示词、近期工具调用和结果 |
| Esc | 返回上一层 |
| 方向键 | 当代理详情内容过长时滚动查看 |
| 空格 | 暂停或恢复运行 |
| `s` | 停止选中的代理；如果焦点在整个工作流上，则停止整个工作流 |
| `r` | 重启选中的运行中代理 |
| `S` | 将这次运行的脚本保存为命令 |

## 让 Claude 编写工作流

你可以用两种方式让 Claude 为任务编写工作流：

1. 在提示词中明确写出 `workflow`，Claude 就会为这个任务生成工作流；
2. 开启 `ultracode`，让 Claude 在当前会话中自动判断哪些实质性任务应该用工作流处理。

你也可以直接运行已有的工作流命令，比如内置的 `/deep-research`，或你之前保存过的自定义工作流。

### 在提示词中请求工作流

如果只想让某一个任务使用工作流，而不改变整个会话的 effort level，只要在提示词中包含 `workflow` 这个词即可。

例如：

```text
Run a workflow to audit every API endpoint under src/routes/ for missing auth checks
```

Claude Code 会高亮你输入里的 `workflow`，然后 Claude 会为这个任务编写工作流脚本，而不是继续用普通的一轮轮对话方式处理。

如果这次运行符合预期，你可以之后把它保存成命令。

如果 Claude Code 高亮了 `workflow`，但你其实并不想触发工作流，可以按 `alt+w`，让本次提示忽略这个触发词。

### 使用 ultracode 让 Claude 自动判断

`ultracode` 是 Claude Code 的一个设置，它把 `xhigh` reasoning effort 和自动工作流编排结合在一起。开启后，Claude 会为当前会话里的每个实质性任务规划工作流，而不是等你显式要求。

开启方式：

```text
/effort ultracode
```

开启 `ultracode` 后，Claude 会自己判断一个任务是否值得启动工作流。一次请求可能会连续拆成多个工作流：一个用于理解代码，一个用于完成修改，一个用于验证结果。

这会应用到当前会话中的每个任务，所以每次请求通常都会比低 effort level 消耗更多 token，也需要更长时间。

`ultracode` 只在当前会话中生效，开启新会话后会重置。回到日常任务时，可以切回：

```text
/effort high
```

它只会出现在支持 `xhigh` effort 的模型上；如果当前模型不支持，`/effort` 菜单里就不会显示这个选项。

## 运行前审批计划

在 CLI 中，每次运行前的提示会展示规划好的阶段，并提供这些选项：

| 选项 | 含义 |
| --- | --- |
| `Yes, run it` | 启动这次运行 |
| `Yes, and don’t ask again for <name> in <path>` | 启动，并且以后在这个项目中运行该工作流时不再询问 |
| `View raw script` | 先查看原始脚本，再决定是否运行 |
| `No` | 取消运行 |

还有两个有用的操作：

- `Ctrl+G`：在编辑器中打开脚本；
- `Tab`：在工作流启动前调整提示词。

你是否会看到这个审批提示，取决于当前权限模式：

| 权限模式 | 何时提示 |
| --- | --- |
| Default、accept edits | 每次运行都会提示，除非你已经为该项目中的这个工作流选择过“不再询问” |
| Auto | 第一次启动时提示。任何一次 `Yes` 都会把同意记录到用户设置中，之后自动启动。开启 `ultracode` 时会跳过提示 |
| Bypass permissions、`claude -p`、Agent SDK | 不提示，工作流会直接启动 |

在 Desktop 应用中，审批卡片会展示工作流名称、阶段列表和 token 用量提醒，并提供 `Once`、`Always` 和 `Deny` 三个操作。进度视图会出现在 Background tasks 侧边栏中。

需要注意：权限模式只控制“是否启动工作流”的这个提示。工作流生成的子代理始终运行在 `acceptEdits` 模式下，并继承你的工具 allowlist；文件编辑会被自动批准。

不过，shell 命令、web fetch 和不在 allowlist 中的 MCP 工具，仍然可能在运行过程中请求你确认。对于长时间运行的工作流，最好在启动前把代理需要用到的命令加入 allowlist。

在 `claude -p` 和 Agent SDK 中，没有交互式确认环节，因此工具调用会直接遵循你配置好的权限规则。

## 保存工作流，方便复用

如果 Claude 为某个你会重复执行的任务写了工作流，可以把这次运行的脚本保存为命令。比如，每个分支都要跑一次的代码审查流程，就很适合保存下来。

保存方法：

1. 运行 `/workflows`；
2. 选择你想保留的运行记录；
3. 按 `S` 打开保存对话框；
4. 用 Tab 在两个保存位置之间切换：
   - `.claude/workflows/`：保存在项目中，克隆仓库的人都能使用；
   - `~/.claude/workflows/`：保存在你的 home 目录中，所有项目可用，但只有你能看到；
5. 按 Enter 保存。

保存后，这个工作流会在之后的会话中以 `/<name>` 的形式运行。

如果项目工作流和个人工作流重名，会优先运行项目中的那个。

## 工作流如何运行

工作流运行时会在一个隔离环境中执行脚本，与当前对话分开。中间结果保存在脚本变量里，而不是不断塞进 Claude 的上下文窗口。

运行时会持续记录每个代理的结果，因此在同一个 Claude Code 会话中，工作流可以恢复运行。

## 行为与限制

工作流运行时有一些约束：

| 约束 | 原因 |
| --- | --- |
| 运行过程中不能要求用户输入 | 只有代理权限提示能暂停运行。如果需要阶段间人工确认，应把每个阶段拆成单独的工作流 |
| 工作流脚本本身不能直接访问文件系统或 shell | 文件读写和命令执行由代理完成，脚本只负责协调代理 |
| 最多 16 个代理并发运行；CPU 核心较少的机器上可能更少 | 控制本地资源占用 |
| 单次运行最多 1,000 个代理 | 防止失控循环 |

## 管理运行中的工作流

工作流启动后，可以从 `/workflows` 视图管理，也可以展开输入框下方任务面板中的进度行来管理。

### 暂停后恢复

如果你停止了某次运行，之后可以恢复它：已经完成的代理会返回缓存结果，剩余代理会继续实时运行。

可以在 `/workflows` 中选择对应运行并按空格恢复，也可以让 Claude 用同一段脚本重新启动工作流。

恢复只在同一个 Claude Code 会话中有效。如果你在工作流运行时退出 Claude Code，下一次会话会从头启动该工作流。

## 成本与用量

工作流会启动许多代理，因此一次运行可能比在普通对话中完成同一任务消耗更多 token。它和其他会话一样，会计入你的方案用量和速率限制。

你可以随时在 `/workflows` 中停止正在运行的工作流，而且不会丢失已经完成的部分。

默认情况下，工作流中的每个代理都会使用当前会话的模型，除非脚本把某些阶段路由到其他模型。想控制模型成本，可以：

- 在大型运行前检查 `/model`，尤其是你平时会切换到较小模型处理日常任务时；
- 描述任务时，明确要求 Claude 在不需要最强模型的阶段使用较小模型。

## 关闭工作流

工作流可用于 CLI、Desktop 应用、IDE 扩展、`claude -p` 非交互模式，以及 Agent SDK。相同的关闭设置适用于所有这些入口。

如果只想为自己关闭工作流，可以选择以下任一方式：

- 在 `/config` 中关闭 **Dynamic workflows**，该设置会跨会话保留；
- 在 `~/.claude/settings.json` 中设置：

```json
{
  "disableWorkflows": true
}
```

- 设置环境变量：

```bash
CLAUDE_CODE_DISABLE_WORKFLOWS=1
```

环境变量会在启动时读取，因此只会影响它被设置到的运行环境。

如果要为整个组织关闭工作流，可以在 managed settings 中设置：

```json
{
  "disableWorkflows": true
}
```

也可以在 Claude Code admin settings 页面中关闭。

当工作流被禁用后：

- 内置工作流命令不可用；
- `workflow` 关键词不会再触发工作流；
- `ultracode` 会从 `/effort` 菜单中移除。
